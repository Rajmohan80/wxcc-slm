# WxCC SLM — System Architecture

**AbhavTech Consulting**  
Version 1.0 · July 2026

---

## Design philosophy

Three principles shaped every architectural decision in this system:

**1. The intelligence is in the workbooks, not the model.**  
A model's weights are opaque, unauditable, and stale on day one. This system encodes consulting knowledge explicitly in structured workbooks with machine-readable rules, selection logic, and auto-flag triggers. The model classifies and formats. The workbooks decide what is correct.

**2. Provenance is a hard requirement, not a feature.**  
Every corpus document is tagged with a provenance tier. Every chunk carries that tier into Qdrant. Retrieval ranks Tier 1 (vendor primary source) above Tier 2 (authored synthesis). Every response carries a `knowledge-current-as-of` stamp, and the n8n automation loop keeps that stamp honest.

**3. A live system beats a complete corpus.**  
The biggest risk is not an incomplete document library — it is a system that never runs. Architecture decisions were made to minimise time-to-first-query: Claude API (no GPU needed), Qdrant Cloud free tier (no infrastructure), Streamlit Cloud (no DevOps). The corpus grows after the pipeline validates.

### Layer 3 — Orchestration (Intent Flow Pipeline)

**LangChain + LangGraph** · `pipeline/slm_pipeline.py`

LangChain builds the agent. LangGraph orchestrates it as a typed state machine. These are distinct roles — LangGraph is not a replacement for LangChain, and MCP is not a replacement for either. The MCP server exposes deterministic tools that the agent calls. The state machine decides when to call them.

#### 9-step pipeline

```
Step 1 — Intent Classifier
    Model: Claude Haiku 4.5 (prompt-cached system prompt)
    Output: intent ∈ {architecture_design, capacity_planning, migration,
                       troubleshooting, compliance_check, feature_query,
                       general_info}
    Cost:   ~$0.0003 per query (cached)

Step 2 — Scenario Detector
    Model: Claude Haiku 4.5
    Output: scenario ∈ {greenfield, migration, HA_DR, security_hardening,
                         AI_enablement, multi_site, compliance_audit}

Step 3 — Requirement Completeness Check
    Source: Workbook A (Required Fields tab, per domain + scenario)
    Output: missing_fields[] — fields the SLM cannot assume

Step 4 — Stop Condition Check
    Source: hard-coded rules + Workbook A (Stop conditions)
    Logic:  runs BEFORE any generation call
    Examples:
      "mainland china" → STOP: not in Cisco data locality tables
      "CUCM only, no cloud" → STOP: out of WxCC scope
      "real-time call recording for minors in DE" → STOP: works council required

Step 5 — Missing Info Gate
    If missing_fields[] is non-empty:
      → return clarification question (exact wording from Workbook A)
      → pipeline terminates; user provides answer; restart from Step 1
    If empty:
      → proceed

Step 6 — RAG Retrieval
    Engine:    Qdrant Cloud (BGE-M3, 1024-dim, cosine)
    top_k:     8 chunks
    Filter:    active=true (never retrieve superseded chunks)
    Ranking:   provenance_tier=1 outranks tier=2 at equal score
    Folder hint: intent→domain→folder mapping for precision
    Fallback:  full-collection search if folder-scoped retrieval returns < 3 chunks

Step 7 — Compliance Flag Injection
    Source:    Workbook B (B12 Data Locality Matrix) + Workbook C (C6 Compliance Patterns)
    Logic:     country-of-operation → DC → applicable regulations
    Output:    compliance_flags[] appended to augmented query before generation

Step 8 — Architecture Generator
    Model:     Claude Sonnet 4.6 (prompt-cached system prompt)
    Input:     query + retrieved chunks + compliance flags + conversation history
    Output:    structured design response with HLD summary, key constraints, assumptions
    Cost:      ~$0.003–0.015 per query depending on response length

Step 9 — Best-Practice Validator
    Source:    Workbook D (Validation Checklist tab, 36 items + auto-check rules)
    Logic:     post-generation pass; flags any generated recommendation that
               contradicts a workbook rule
    Output:    appended warnings if triggered; response stands but is flagged
```

### Layer 4 — Knowledge

#### Workbooks (structured intelligence)

| Workbook | Tabs | Rows | Primary role |
|---|---|---|---|
| **A — Requirements** | 6 | 348 | What to collect before designing |
| **B — Product Knowledge** | 11 | 478 | What the products can and cannot do |
| **C — Architecture** | 10 | 254 | How to design correctly |
| **D — Engineering** | 11 | 414 | How to size, validate, and de-risk |

Machine-readable fields in workbook rows:
- `SLM_Selection_Rule` — logic the pipeline uses to choose between options
- `Auto_Flag_Trigger` — conditions that raise a compliance or risk flag automatically
- `Auto_Check_Rule` — validation rule applied by Step 9

#### RAG corpus (retrieved intelligence)

48 source documents in 14 folders under `webex_slm_corpus/`.

Ingest: `4_ingest_corpus.py` using BGE-M3, chunk size 512 tokens, overlap 64 tokens.

Qdrant chunk metadata schema:

```json
{
  "doc_id": "CI-16",
  "filename": "cisco-wxcc-data-locality.md",
  "folder": "02_wxcc_architecture",
  "provenance_tier": 1,
  "source_url": "https://help.webex.com/en-us/article/n0p6xa1/",
  "source_version": "a3383b56d2a89d15",
  "ingested_at": "2026-07-22T08:14:00Z",
  "active": true,
  "supersedes": null,
  "embed_model": "BAAI/bge-m3",
  "embed_model_version": "1.0"
}
```

Re-ingesting a changed document sets `active=false` on old chunks and creates new ones with `supersedes` pointing back. Old chunks are never hard-deleted — this is the Phase 7 compatible model from day one.

### Layer 5 — MCP Server

**FastMCP** · `mcp_server/mcp_server.py`

11 tools exposed over MCP for A2A integration:

| Tool | Description |
|---|---|
| `search_docs` | Full-text search across corpus with provenance filter |
| `search_vector_db` | Semantic search against Qdrant |
| `capacity_calculator` | Agent/IVR/PSTN capacity from Workbook D formulas |
| `generate_hld` | High-level design document from requirements |
| `generate_lld` | Low-level design from HLD + product knowledge |
| `export_pdf` | Render design output as downloadable PDF |
| `risk_register` | Pre-populated risk register from Workbook D (32 risks) |
| `diagnose_ivr` | IVR fault diagnosis using Workbook D decision trees |
| `check_compliance` | Compliance pattern lookup from Workbook C |
| `data_locality` | Country → DC → regulation mapping from B12 |
| `feature_parity` | WxCC vs competitor feature comparison from B11 |

The MCP server is the integration point for Webex AI Agent Studio (A2A). An enterprise customer's AI orchestrator can delegate WxCC-specific design queries to this SLM agent via MCP, receiving structured design artifacts in return.

### Layer 6 — Automation (in pipeline)

**n8n · 4-workflow knowledge currency loop**

Full specification: [docs/phase7_n8n_spec.md](phase7_n8n_spec.md)

Summary:

```
Workflow 1: Source Watcher    — SHA-256 diff of live Cisco/GCP/regulator pages (weekly)
Workflow 2: Change Triage     — Haiku classifies each change as CRITICAL/MAJOR/MINOR
Workflow 3: Apply & Re-index  — human-gated corpus update + Qdrant supersede upsert
Workflow 4: Golden Test Set   — weekly regression run; flags retrieval degradation
```

The runtime pipeline never checks the web during a query. All knowledge currency checks run offline on the n8n schedule. Deliverables carry a `knowledge-current-as-of` footer stamp.

## Stack decisions log

| Decision | Choice | Rejected | Rationale |
|---|---|---|---|
| Classification LLM | Claude Haiku 4.5 | GPT-4o-mini, Groq/Llama | Prompt caching economics; no GPU required; consistent JSON output |
| Generation LLM | Claude Sonnet 4.6 | GPT-4o, Gemini 2.5 Flash | Quality/cost balance; single-vendor for prompt cache; simpler credential management |
| Embedding | BGE-M3 | all-MiniLM-L6-v2 | 1024-dim outperforms 384-dim on technical multilingual queries; MTEB benchmark |
| Vector DB | Qdrant Cloud | Pinecone, ChromaDB | Free tier generous; `active` flag native in metadata; consistent Python API |
| Orchestration | LangChain + LangGraph | pure LangGraph, CrewAI | LangChain agent + LangGraph state machine = correct separation of concerns |
| MCP | FastMCP | custom REST | Portfolio showcase for MCP/A2A architecture pattern; minimal overhead |
| Deployment | GCP Cloud Run | AWS Lambda, Azure Functions | $300 GCP credits; Python-native; scales to zero |
| V2 LLM | Qwen3 8B Instruct | Mistral 7B | Better reasoning on structured consulting tasks; open-weight; QLoRA-ready |
| V2 trigger | Enterprise demand | build schedule | V2 is a deployment model, not an upgrade. Triggered when a customer needs data sovereignty. |
